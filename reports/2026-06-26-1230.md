## #2186 root cause CORRECTED — important

Earlier theory ("the UI's fire-and-forget `registerTransaction` POST to mana was lost") is **WRONG**. Wan supplied the contradicting fact: the vote **modal showed SUCCESS** in every missing-vote case.

- The modal only flips to `success` **after `await registerTransaction` resolves** (failure re-throws → modal `fail`; `apps/ui-*` `useActions.ts:82`, `mana.ts:15-33`).
- So success means the POST returned 200 **and** the `registered_transactions` row WAS created (`rpc.ts:92-94`).
- Therefore the bug is **RELAYER-SIDE**, not lost-POST.

### The actual mechanism (mana 24h sweep)
- `apps/mana/src/stark/registered.ts:86` calls `markOldTransactionsAsProcessed()` **every loop**.
- `apps/mana/src/db.ts:47-51` unconditionally force-sets `processed=true, failed=true` on **any** row older than 24h — even if its EthTx L1→L2 commit hasn't been consumed yet.
- The `_commits == 0x0 → return` early-exit (`registered.ts:33`) leaves the row unprotected while the commit is still in flight. The 24h sweep retires it; the commit later lands (`_commits=0x1`) but the now-`failed` row is **never retried** → commit stuck at 0x1, vote never indexed.
- **Hash-recompute theory RULED OUT**: the relayer reuses the stored hash+payload, no recompute.

### Corrected fix direction
- Never auto-expire a row whose on-chain `_commits == 0x1` (keep retrying — it's recoverable).
- Only retire rows still at `0x0` past a window **beyond worst-case L1→L2 latency**.
- Alert instead of silently dropping after 3 `client.vote` failures (`registered.ts:62-73`).
- Secondary silent-drop path to watch: `db.ts:22-32` `registerTransaction` uses `onConflict().ignore()` on `unique(sender, hash)`.
