## #2186 missing-votes ‚Äî FINAL root cause (corrected twice, now evidence-backed)

The previous TWO theories are both **DISPROVEN**:

1. **NOT the mana 24h sweep** (`markOldTransactionsAsProcessed`). Disproven by read-only DB: recent **mainnet** rows succeeded, only **old Sepolia** rows failed ‚Äî the failure does not track the 24h boundary.
2. **NOT hash zero-padding.** Disproven: pedersen `getStorageVarAddress` is padding-agnostic ‚Äî `addr('0x291‚Ä¶') === addr('0x0291‚Ä¶')`. Leading-zero variants resolve to the same storage slot.

### ACTUAL cause ‚Äî a lost / non-durable write at the DB gateway
For the two missing votes, the `registerTransaction` INSERT **never persisted**, yet mana logged "Registering transaction" with **no error** and returned **success**.

Evidence (three independent sources reconciled):
- **Better Stack `mana` logs**: the POST arrived and was logged ("Registering transaction"), no error.
- **Read-only mana DB**: no row exists for the two missing votes.
- **Decisive contrast in the processing loop**: the vote that LANDED (row **id 38**) shows the loop count flip `0‚Üí1` then "broadcasted successfully". The two missing votes **NEVER reached count:1** ‚Üí the write itself never committed, so the processing loop never saw a row to advance.

Mechanism: a lost / non-durable write at the PlanetScale `psdb.cloud` Postgres gateway, **MASKED** by two layers:
- `db.ts` `registerTransaction` uses `.onConflict().ignore()` ‚Äî swallows the no-op.
- `rpc.ts` returns `rpcSuccess(true)` **unconditionally**, with no read-back of the row.

Result: UI sees ‚úÖ (modal success is real ‚Äî the POST returned 200), relayer never sees the row, on-chain commit stuck at `_commits=0x1`.

### Key on-chain insight (corrects an earlier misread)
A **SUCCESSFUL** EthTx vote **CLEARS** `_commits` back to `0x0` ‚Äî the authenticator deletes the commit on consumption. So:
- `_commits = 0x1` = committed-but-never-voted (stuck).
- `_commits = 0x0` **after** a confirmed index = success (NOT "never registered").
Earlier reports read `0x0` as "never registered"; that was wrong for the indexed case.
