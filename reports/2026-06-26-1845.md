A sequence advances on `nextval` regardless of whether the INSERT commits, so persistent gaps = **inserts that allocated an id but never committed**. ## Discussion with Wan — refined the fix (detection ≠ recovery)
Wan pushed for the real root cause ("why is the insert failing", "dig the db logs"). Key reframe: **PR #2189 makes the failure DETECTABLE, not RECOVERED.** Fix ladder:
1. **Client/UI retry** on the `rpcError` — safe + idempotent now that the write is an upsert (`.merge()`).
2. **Relayer self-heal (recommended real fix):** mana watches `StarknetCommit` `CommitAdded` / scans `_commits` and self-registers committed-but-unregistered votes independent of the client POST. Recovers even orphaned commits — the robust fix.
3. **Write durability:** knex pool `afterCreate` ping + wrap the insert in a txn so a dropped psdb connection **rejects** instead of silently no-op'ing.
- Recommended option 2 to Wan as the real fix; offered to build it as a follow-up.
